# [SPEC-003]: Phase 2b — 內容機制擴充（消耗品 / 奇遇連鎖 / 門派 / 稱號）

| 欄位 | 內容 |
|------|------|
| **狀態** | `Draft`（依賴 ADR-001 FIRM；含 schema v3 變更）|
| **日期** | 2026-07-21 |
| **關聯** | ADR-001（技術棧+敘事層）、SPEC-002（成長系統，本 SPEC 於其上擴充）|

---

## Goal（這個功能建了什麼）

在既有成長系統上加入四項內容機制，讓江湖更耐玩：**消耗品/丹藥**（購買並使用，回復或增益）、**奇遇連鎖劇情**（多步驟分支事件，選擇影響結果）、**更多門派**（含專屬功法）、**稱號/成就**（依成就條件解鎖，給長期目標）。存檔升至 v3 並自動遷移 v1/v2。

## Inputs（什麼資料/事件進入系統）

- **玩家操作**：使用消耗品、於奇遇中做選擇、（建角時可選新門派）。格式：UI 事件。
- **新增資料表**：`consumables.ts`（丹藥）、`encounters.ts`（連鎖劇情）、`titles.ts`（稱號/成就）；擴充 `sects.ts`、`skills.ts`。
- **既有存檔**：啟動讀 localStorage；v1/v2 → 遷移。

## Outputs（系統產生什麼輸出）

- **成功路徑**：
  - 消耗品：使用後即時回氣血/體力、或永久加屬性，數量遞減；用罄自背包移除。（暫時增益見 Non-Goals）
  - 奇遇：呈現情境與 2–3 個選項；選擇後結算對應結果（獎懲/戰鬥/道具）並產生敘事。
  - 門派：建角可選新門派，其專屬功法在功法面板可學。
  - 稱號：達成條件時解鎖並提示；已解鎖稱號可於角色資訊顯示。
- **失敗路徑**：
  - 使用不存在/數量為 0 的消耗品 → 拒絕、提示。
  - 奇遇選項需要資源（銀兩/等級）不足 → 該選項禁用或拒絕。
  - 遷移失敗（結構無法識別）→ 保留原存檔、提示。

## Side Effects（狀態變更、外部呼叫、通知）

- **Character schema 擴充**：新增 `consumables`（{itemId: count} 計數映射）、`titles`（已解鎖稱號 id 陣列）。
- **存檔 schema v2 → v3**：`SAVE_VERSION = 3`；`migrateCharacter` 補新欄位預設（空消耗品、空稱號）。維持「只補不刪、冪等」。
- **奇遇待決狀態**：store 增加 `pendingEncounter`（目前進行中的連鎖劇情節點）。
- **無外部網路呼叫**（Narrator 仍 Template）。

## Edge Cases（錯誤條件、邊界輸入、race condition）

- v1/v2 存檔遷移至 v3：舊進度保留，新欄位補預設；冪等。
- 消耗品數量邊界：使用最後一個後移除；數量不得為負。
- 奇遇進行中重整頁面：pendingEncounter 若不持久化，重整後應安全回到主畫面（不卡死）。
- 奇遇選項指向不存在的下一節點 → 安全結束奇遇，不 crash。
- 稱號重複解鎖 → 冪等，不重複加入。
- 使用永久加屬性丹藥 → 屬性上限/氣血同步重算（若影響 maxHp）。
- 空資料表（無消耗品/無稱號）→ UI 顯示「暫無」，不 crash。

## Done When（二元可測試驗收條件）

- [ ] 可購買消耗品並使用；回氣血類即時生效、永久加屬性類永久生效，數量正確遞減（純函式 + 測試）。
- [ ] 使用數量為 0 或不存在的消耗品被拒（測試覆蓋）。
- [ ] 至少一個奇遇連鎖：呈現選項 → 選擇 → 對應結果，且不同選擇導致不同結果（純函式解析 + 測試）。
- [ ] 奇遇選項資源不足時被正確處理（禁用/拒絕，測試覆蓋）。
- [ ] 新增門派 ≥2（總數 ≥6），各有專屬功法；外派仍不可學其專屬（沿用 SPEC-002 規則，測試覆蓋新門派）。
- [ ] 稱號系統：依條件（如等級達標、擊敗特定 BOSS、聲望門檻）解鎖至少 4 個稱號；解鎖判定為純函式且冪等（測試覆蓋）。
- [ ] 存檔 v2→v3 自動遷移並正常遊玩；`migrateCharacter` 測試含 v2→v3（新欄位補預設、冪等）。
- [ ] `npm test` 全綠、`npm run typecheck` exit 0、核心模組 coverage ≥ 80%。
- [ ] 真實瀏覽器 E2E：使用消耗品、完成一段奇遇選擇、（顯示稱號）可實際操作。

## Non-Goals（明確不做）

- 不接真實 AI（Phase 3）。
- 消耗品不做合成/煉丹小遊戲（先為現成道具）。
- 消耗品不做「暫時增益」（activeBuffs，含回合過期追蹤）；本 SPEC 僅即時回復與永久加屬性兩類。暫時增益留待後續。
- 奇遇不做跨場記憶的長篇任務線（先為單段 2–3 步分支）。
- 稱號不做數值加成（先為榮譽展示；加成可後續）。
- 不做後端、多人。

## Rollback Plan

引入存檔 schema v3（migration）。回滾策略同 SPEC-002：
1. 程式回滾 = `git revert`。
2. migration 只補欄位不刪 v1/v2 既有欄位 → 回滾到舊版仍可讀舊欄位（新欄位被忽略）。
3. v3 存檔在只認 v2 的舊程式會被「版本較新」邏輯拒絕載入（既有保護），不損毀。
4. 奇遇待決狀態（pendingEncounter）預設不持久化，避免版本間半殘狀態。

## 實作順序（依賴驅動，非驗收條件）

1. schema v3 + 遷移（地基）。
2. 消耗品：資料 + 使用純函式 + 測試 + store action + UI。
3. 稱號：資料 + 解鎖判定純函式 + 測試 + 接入行動後檢查 + UI 顯示。
4. 更多門派 + 專屬功法（純資料）。
5. 奇遇連鎖：分支資料結構 + 解析純函式 + 測試 + pendingEncounter store + 選擇 modal UI。
6. E2E 擴充驗證。
