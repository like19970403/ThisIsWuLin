# [SPEC-002]: Phase 2 — 成長系統（等級 / 功法 / 裝備 / 內容）

| 欄位 | 內容 |
|------|------|
| **狀態** | `Draft`（依賴 ADR-001 FIRM；含 schema 變更）|
| **日期** | 2026-07-21 |
| **關聯 ADR** | ADR-001（技術棧+敘事層）；本 SPEC 引入存檔 schema v2，屬 ADR-001「存檔版本化」後續追蹤項 |

---

## Goal（這個功能建了什麼）

為「這就是江湖」加入**角色成長閉環**：練功與戰鬥獲得經驗 → 升級提升屬性上限並解鎖學功法資格 → 消耗資源學習功法並裝備上陣（戰鬥時發動效果）→ 闖蕩/商店取得裝備穿戴提升戰力 → 更豐富的事件/敵人/門派支撐這個循環。存檔升級至 v2 並自動遷移舊 v1 存檔。

## Inputs（什麼資料/事件進入系統）

- **玩家操作**：學功法（在功法面板點選）、裝備/卸下功法與裝備、購買裝備（商店事件）。格式：UI 事件。
- **新增資料表**（靜態，TS/JSON）：功法表 `skills.ts`、裝備表 `equipment.ts`；擴充 `events.ts`（商店/奇遇/學功法機會）、`enemies.ts`、`sects.ts`。
- **既有存檔**：啟動時讀 localStorage；若為 v1 → 觸發遷移。

## Outputs（系統產生什麼輸出）

- **成功路徑**：
  - 經驗條、等級顯示於屬性面板；升級時產生敘事與提示。
  - 功法面板：已學/未學功法、可裝備欄位；裝備面板：武器/防具欄 + 背包。
  - 戰鬥結算納入「已裝備功法效果」與「裝備屬性加成」，回合日誌反映功法發動。
  - 商店事件可用銀兩購買裝備。
- **失敗路徑**：
  - 資源不足（等級/銀兩不夠）學功法/買裝備 → 拒絕 + 提示，不扣資源。
  - 裝備欄已滿仍裝備 → 替換舊裝備並回背包（或提示）。
  - 舊存檔遷移失敗（結構無法識別）→ 保留原存檔、提示、不套用半殘資料。

## Side Effects（狀態變更、外部呼叫、通知）

- **Character schema 擴充**：新增 `level`、`exp`、`learnedSkillIds`、`equippedSkillIds`、`equipment`（weapon/armor 欄）、`inventory`（道具/裝備 id 陣列）。
- **存檔 schema v1 → v2**：`SAVE_VERSION = 2`；`migrateSave(v1) → v2` 補預設值（level 1、exp 0、空功法/裝備/背包）。
- **localStorage 寫入** key 不變（`wulin-save-v1` 的 store name 保留，內部 version 欄位改 2；載入時依 version 決定是否遷移）。
- **無外部網路呼叫**（Narrator 仍為 Template；AI 屬 Phase 3）。

## Edge Cases（錯誤條件、邊界輸入、race condition）

- 舊 v1 存檔載入 → 自動遷移，玩家進度（姓名/屬性/銀兩/聲望）完整保留，新欄位補預設。
- 遷移須冪等：對已是 v2 的存檔再跑 migrate 不得破壞資料。
- 經驗溢位/負值、等級上限 → clamp。
- 學已學過的功法 → 拒絕（不重複扣資源）。
- 裝備不存在的 id（資料表移除過的舊裝備）→ 忽略該 id，不 crash。
- 裝備功法數超過上限 → 拒絕或替換，不得超欄。
- 戰鬥中功法效果數值 → 全部經 clamp，氣血/傷害不為負（延續 Phase 0 保證）。
- 空功法表/空裝備表 → 面板顯示「暫無」，遊戲不 crash。
- 存檔資料損毀（localStorage JSON parse 失敗、或關鍵欄位型別非預期如 `character` 非物件）→ 視為無存檔、導向建角，不套用半殘資料、不 crash。

## Done When（二元可測試驗收條件）

- [ ] 練功與戰鬥勝利獲得經驗；經驗達門檻自動升級，等級+1、屬性上限提升，UI 反映。
- [ ] 升級純函式（`gainExp` / `levelUp`）有單元測試，邊界（剛好升級、連升多級、溢位）覆蓋。
- [ ] 可在功法面板學習一個符合資格的功法；資源不足時被拒且不扣資源（測試覆蓋）。
- [ ] 裝備一個功法後，戰鬥結算數值改變（有功法 vs 無功法結果不同），且為純函式可測試。
- [ ] 可取得並穿戴一件裝備，屬性/戰力隨之提升，卸下後還原（測試覆蓋）。
- [ ] 商店事件可用銀兩購買裝備；銀兩不足被拒（測試覆蓋）。
- [ ] 內容擴充：功法 ≥6、裝備 ≥6、事件總數 ≥12、敵人 ≥6、門派 ≥4（資料表計數可驗）。
- [ ] 舊 v1 存檔可自動遷移至 v2 並正常遊玩；`migrateSave` 有單元測試（含冪等性）。
- [ ] `npm test` 全綠、`npm run typecheck` exit 0、核心模組 coverage ≥ 80%。
- [ ] 真實瀏覽器 E2E：學功法→裝備→戰鬥發動→升級→穿裝備 這條垂直切片可實際操作（browser-check 擴充後通過）。

## Non-Goals（Phase 2 明確不做）

- 不接真實 AI（Phase 3）。
- 不做多人、交易、拍賣行。
- 不做複雜技能樹/連招 combo（功法先為「被動加成 + 單一發動效果」等級）。
- 不做裝備強化/鑲嵌/耐久（先為固定屬性裝備）。
- 不做後端、帳號、雲存檔。

## Rollback Plan

本 Phase 引入存檔 schema v2（有 migration）。回滾策略：
1. 程式回滾 = `git revert` 對應 commit。
2. **存檔相容**：v2 存檔在回滾到只認 v1 的舊程式時會被「版本不符」邏輯拒絕載入（既有保護），不會損毀；但玩家 Phase 2 進度會回不去。故 migration 只單向補欄位、**不刪除** v1 既有欄位，確保回滾後 v1 欄位仍完整可讀。
3. 上線前（Phase 3 之後才談上線）需在 release note 標明存檔版本升級。

## 實作順序（依賴驅動，供 ROADMAP 參考，非驗收條件）

1. **schema + 遷移**：擴充 Character 型別、`SAVE_VERSION=2`、`migrateSave` + 測試（地基，先做以免後續改存檔反覆）。
2. **等級/經驗**：`gainExp`/`levelUp` 純函式 + 測試 → 接入 train/戰鬥勝利。
3. **功法系統**：`skills.ts` 資料 + 學習/裝備邏輯 + 戰鬥結算納入功法效果 + 測試。
4. **裝備系統**：`equipment.ts` 資料 + 穿戴/卸下 + 屬性加成 + 測試。
5. **內容擴充**：補足事件（含商店）、敵人、門派至 Done When 數量門檻。
6. **UI**：功法面板、裝備面板、經驗/等級顯示、商店互動。
7. **E2E**：擴充 browser-check 覆蓋垂直切片。
